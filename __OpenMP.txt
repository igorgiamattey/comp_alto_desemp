OpenMP

1. Conceito Geral OpenMP (Open Multi-Processing)
	é uma API utilizada para programação paralela em sistemas de memória compartilhada. Ela permite dividir um programa em múltiplas threads que executam simultaneamente, aproveitando melhor múltiplos núcleos do processador.

	Processo vs Thread:
		- Processo: programa em execução com seu próprio espaço de memória. Processos isolados não se comunicam facilmente.
		- Thread: unidade leve dentro de um processo. Threads compartilham memória, o que facilita comunicação, mas exige cuidado com concorrência.

2. Diretivas (#pragma omp)
	As diretivas OpenMP são instruções especiais para o compilador. Elas indicam quais partes do código devem ser executadas em paralelo. O compilador ignora essas linhas se a flag de compilação do OpenMP não for ativada.

	Exemplo:
	#pragma omp parallel
	{
		// código executado por múltiplas threads
	}

3. Modelo Fork-Join
	- Inicialmente existe apenas uma thread (master)
	- Fork: cria várias threads
	- Execução paralela
	- Join: threads finalizam e voltam para uma única

4. Sincronização
	- Barrier: todas as threads esperam até que todas cheguem nesse ponto.
	- Critical: apenas uma thread executa um bloco por vez. É útil para blocos de código mais longos, mas gera "gargalos" de desempenho porque enfileira as threads.
	- Atomic: garante que uma operação simples seja feita sem interferência. É muito mais rápido que o "critical" porque utiliza instruções de hardware, mas só serve para operações como "x++" ou "x += 5".

5. Escopo de Variáveis
	- Private: cada thread tem sua própria cópia não inicializada da variável.
		# firstprivate: inicializa com valor original

		'''
		int x = 50

		#pragma omp parallel firstprivate(x)
		{
			int id = omp_get_thread_num();
			x = x + (id * 10);
			// aqui temos o valor alterado para cada thread
			printf("x = %d\n", x); // x = 50 + (10 * id)
		}

		// aqui teremos o valor original de x, sem nenhuma alteração
		printf("x = %d\n", x); // x = 50
		'''

		# lastprivate: mantém valor final da última iteração

		'''
		int x = 0;

		#pragma omp parallel for lastprivate(x)
		for (int i = 0; i <= 50; i++) {
			
			// aqui x foi inicializado com lixo de memória
			printf("x = %d\n", x); // x = lixo de memória
			x = i * 2;

			// aqui x recebe o valor calculado na iteração atual
			printf("x = %d\n", x); // x = 2*i
		}
		
		// aqui x recebe o valor calculado na última iteração do loop, não necessariamente na última thread que finalizou, portanto recebe o valor com i = 50
		printf("x = %d\n", x); // x = 2*50 = 100
		'''
	- Shared: A variável é compartilhada entre todas as threads (padrão para variáveis declaradas antes do bloco paralelo). Modificações afetam todas as threads.

6. Controle de Execução
	- Master: só thread principal executa, as outras threads passam direto, sem esperar
	- Single: apenas uma thread qualquer executa, as outras threads esperam até ela terminar.

7. Parallel For
	- divide iterações de loops entre threads automaticamente.

8. Reduction
	- Permite combinar resultados de múltiplas threads (ex: soma, produto).
	- #pragma omp parallel for reduction(operation:variable)
		# operation: +, -, *, /, ...
		# variable: a variável em que se deseja realizar tal operação

9. Chunks
	- Define o tamanho dos blocos distribuídos para cada thread.

10. Schedule
	- Define como o trabalho é distribuído:
		# Static: divisão fixa
		# Dynamic: threads recebem tarefas dinamicamente. Se uma thread termina seu chunk, ela pede mais.
		# Guided: blocos grandes no início e menores depois

11. Ordered e Depend
	- Ordered: mantém ordem de execução
	- Depend: Controla dependência entre tarefas
		# `depend(out: x)` indica que uma tarefa gera o resultado (source).
    	# `depend(in: x)` indica que uma tarefa posterior precisa aguardar esse valor (sink).

12. Sections (Paralelismo de Tarefas Distintas)

	A diretiva #pragma omp sections permite que você divida o trabalho em blocos de código independentes. O OpenMP garante que cada bloco (seção) seja executado exatamente uma vez por uma única thread da equipe.

		A Analogia da Cozinha:

			parallel for: Você tem 100 cebolas para picar e 4 cozinheiros. Todos os 4 cozinheiros pegam facas e picam cebolas juntos (mesma tarefa, dados divididos).

			sections: O cliente pediu um prato complexo. Um cozinheiro grelha a carne, outro frita a batata e um terceiro prepara a salada. (Tarefas diferentes executadas simultaneamente).

		Sections vs Tasks (Qual usar?)

			Você viu as tasks no exemplo da lista encadeada (linked list) anteriormente. Ambas servem para paralelismo de tarefas, mas:

				Use sections quando você tem um número fixo, pequeno e bem conhecido de tarefas independentes que você já sabe exatamente quais são antes de rodar o código (ex: Inicializar 4 módulos diferentes de um sistema). É estático.

				Use tasks quando você tem um número desconhecido de tarefas geradas dinamicamente durante a execução (ex: percorrer uma lista encadeada, algoritmos recursivos como o Quicksort).

13. Paralelismo Aninhado (Nested Parallelism)

	Ocorre quando uma thread que já está executando dentro de uma região paralela encontra uma nova diretiva #pragma omp parallel.

		A Analogia da Subcontratação:
		Imagine que você é o gerente (Thread Mestre) e contrata 4 mestres de obras (Nível 1). O paralelismo aninhado permite que cada um desses mestres de obras contrate sua própria equipe de 4 pedreiros (Nível 2) para realizar uma subtarefa.

	O Comportamento Padrão (Desativado)

	Por padrão, o OpenMP desativa o paralelismo aninhado por motivos de segurança.
	Se você colocar um parallel dentro de outro sem avisar o compilador, a thread do nível 1 criará uma "equipe de 1 pessoa" (ela mesma) e executará o bloco interno de forma sequencial.

	Para fazer funcionar de verdade, você precisa ligar essa chave explicitamente no código usando omp_set_nested(1) (ou em versões mais recentes do OpenMP, omp_set_max_active_levels()).

14. Collapse (Achatamento de Loops Aninhados)

	A cláusula collapse(N) é usada junto com o #pragma omp parallel for para transformar (achatar) N loops aninhados em um único loop gigante. Em seguida, o OpenMP divide automaticamente esse novo "super loop" entre as threads disponíveis.

		O Problema que ele resolve: Imagine que você tem uma matriz de 3 x 1000 e um processador de 8 núcleos. Se você colocar um parallel for apenas no loop externo (que vai até 3), apenas 3 threads vão trabalhar e 5 ficarão ociosas. O collapse resolve isso sem precisar de paralelismo aninhado (que causaria explosão de threads).

		A Analogia do Tabuleiro:
		Em vez de distribuir "linhas inteiras" do tabuleiro para cada pessoa, você transforma o tabuleiro em uma fita longa e contínua de quadradinhos e distribui fatias iguais dessa fita para todos os trabalhadores.

	Como funciona a sintaxe

	Você indica o número de loops que deseja achatar dentro dos parênteses: collapse(2) para achatar dois loops, collapse(3) para três, etc.

	Regra de Ouro (Loops Perfeitamente Aninhados)

	Para que o collapse funcione, os loops precisam ser perfeitamente aninhados. Isso significa que não pode haver nenhuma linha de código (como printf ou inicialização de variáveis que não fazem parte do loop) entre o primeiro for e o segundo for.

		ERRADO:
		#pragma omp parallel for collapse(2)
		for (int i = 0; i < 10; i++) {
			printf("Nova linha\n"); // PROIBIDO: Quebra o aninhamento perfeito!
			for (int j = 0; j < 10; j++) { ... }
		}

		CORRETO:
		#pragma omp parallel for collapse(2)
		for (int i = 0; i < 10; i++) {
			for (int j = 0; j < 10; j++) { ... } // Nada entre os 'for's
		}
